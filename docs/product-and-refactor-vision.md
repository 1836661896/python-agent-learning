## 1. 本文档只回答什么

- **为何重构**、**工程原则**、**同仓库如何迁移/重写 `src`** 等与「怎么把代码摆好」相关的共识。  
- 与 **`docs/project-goals.md`** 冲突时，以 **产品目标文档** 为准；与 **`backend-refactor-plan.md`** 冲突时，以 **最近 git 更新** 或 **用户明确更正** 为准。

---

## 2. 重构动机与工程原则

| 痛点 | 原则 |
|------|------|
| 项目乱、阅读成本高 | 分层清晰、命名一致、从入口到领域一条线可读 |
| **学习代码与生产代码混杂** | **练习/实验**与 **`src/` 产品代码**分区；实现以 **长期真实使用** 为准 |
| 目录边界不清 | 按 **领域** 竖切（会话、工作流/任务、检索与下载、工具/MCP、LLM 适配等） |
| 重复代码多 | 抽取共用能力（重试与日志、HTTP 客户端、任务状态、工具注册等），路由层保持薄 |

**改动幅度**：接受 **完整重构与阵痛期**；结果须符合 **`project-goals.md`** 与当前 **`readme.md`** 所载模块与接口约定。迁移建议采用 **可运行的小步或模块化搬迁**，避免长期不可运行。

---

## 3. 同仓库「参考旧代码、新建 `src`」策略

- **仓库**：不新建远程仓库，在**当前仓库**内完成。  
- **推荐**：将现有 **`src` 重命名为备份目录**（如 **`src1`**，仅作对照），再 **新建干净的 `src`**，使 **`uvicorn src.api:app`** 与 pytest 路径习惯不变；若可运行包名改为 `src1`，需全局改入口与 import，**非必要不推荐**。  
- **已定案（本仓库）**：备份目录已移除，**旧实现仅以 git 历史可查**；后续扩展一律在现行 **`src/`** 内按 **`backend-refactor-plan.md`** 推进。  
- **数据**：库表与 Alembic 是否复用、是否大改，动刀前列 **迁移清单**。

# 项目目标与预期能力

> **用途**：记录**中长期要建成什么**；安排学习计划、评估需求时**优先阅读本文**。  
> **更新频率**：**低**——仅在提出**新的期望功能**或**对目标有重大调整**时修改；日常迭代改代码不必改本文件。  
> **与 `readme.md` 分工**：`readme.md` 描述**当前代码已具备什么**；本文件描述**希望最终具备什么**（可包含尚未实现的条目）。

---

## 1. 使用者与边界

- **使用者**：现阶段以**本人**为主；未来可扩展到**周围少数人**。多账号、数据隔离、云端数据库等**上云与多用户阶段**再纳入目标。
- **合规与产品边界**：实现采集、下载、游戏自动化等能力时，需单独维护「禁区 / 二次确认 / 白名单」等条款（根目录 `readme.md` 可保留与当前代码一致的**简短合规提示**；细则随功能落地补充）。

---

## 2. 能力愿景（分阶段）

### 2.1 第一阶段（优先）

- **联网检索**：按自然语言需求自动查找资料。  
- **整合与交付**：汇总为可用产物，支持 **文档 / 视频 / 文案 / 素材** 等 **可下载或可导出** 形态。  
- **客户端顺序**：**先做浏览器**；再 **桌面应用 > 移动端**；能力尽量经 **HTTP API / SSE** 暴露，便于换壳。

### 2.2 第二阶段（必达，与自动化主线一致）

- **读屏与识别**：基于屏幕内容做判断。  
- **日常任务自动化**：固定流程可重复执行。  
- **游戏内场景**：固定逻辑执行；进阶到对动态内容的反应（如躲避类），对延迟与稳定性要求高。

---

## 3. 可验收的成功标准（备忘）

1. 描述资料需求 → 自动检索并整合 → **可下载文档**（宜带来源/引用以便核对）。  
2. 指定某类视频需求 → 在能力范围内 **获取或生成** 视频及文案，并约定交付形态。  
3. 本机侧：基于屏幕内容执行固定逻辑，并具备向实时决策演进的空间。

---

## 4. 任务形态与执行策略（目标态）

- **可保存、可重复执行**的工作流或脚本。  
- **定时 / 计划触发**。  
- **人机混合**：下载超体积/数量阈值、游戏内资源消耗与充值等场景 **须先确认**。  
- **失败策略**：有限次 **自动重试** + **完整可追溯日志**。

---

## 5. 长耗时与异步（目标态）

- 提交后返回 **任务 ID**，后台执行；前端通过 **轮询或 SSE** 获取进度与结果。  
- 不默认假设上 Celery；单机内不足再演进到队列/多进程。

---

## 6. 架构与工具（目标态）

- **先单体**：单进程内模块边界清晰；保留未来拆服务的可能。  
- **工具**：内置与 MCP **混合**；学习阶段先掌握内置 tool 调用链；**调用面长期统一为 MCP**（含自研 MCP Server）。

---

## 7. 会话与记忆（目标态）

- **单会话**：消息持久化；超长时 **自动精炼** 以节约 token。  
- **新会话**：不默认携带旧会话全文。  
- **跨会话**：会话列表、检索与查找、历史会话查看；语义级搜索可与向量 / pgvector 等后续结合。

---

## 8. 大模型策略（目标态）

- **主线**：**云端大模型**，支持 **多厂商切换**。  
- **本地（如 Ollama）**：**兜底**，异常时仍可用。

---

## 9. 测试与质量（目标态）

- **pytest** 以 **关键路径** 为满足，随核心能力扩展用例。

---

## 10. 部署与数据（当前共识）

- **现阶段**：**本机为主**；数据库与应用 **同机** 即可。  
- **外网暴露、多用户**：后续再纳入目标。
